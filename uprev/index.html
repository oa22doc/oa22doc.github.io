Automatically Updating 2.0.x Libraries to 2.2

The oa20to22 translator lets you create an OpenAccess 2.2 design library from an OpenAccess 2.0.x design library.

Note: On Windows, make sure that the Cadence cdsCommon DLLs are in your path. (The cdsCommon DLLs are typically included in <install_dir>/tools/bin.) This is not required on the UNIX platforms.

Important: Due to the unique nature of 2.0 to 2.2 translation, oa20to22 makes direct calls to the OpenAccess 2.0 and 2.2 implementation. Developers are strongly discouraged from using such calls, as they are not guaranteed to be consistent from one release to the next.

Usage:
oa20to22 -srcCdsLib path -lib library -libPath path -DMSystem oaDMFileSys -log file

Required Arguments
-srcCdsLib path cds.lib file that contains the definition of the 2.0 library to translate
-lib library Source and destination library name.
Optional Arguments
-dataModel dataModel value Specifies a dataModel value of 0, 1, 2, or 3, which determines the data model for the OpenAccess database and determines how properties are mapped to the database. If this option is omitted, the default dataModel value of 1 is used unless the OA_DATAMODEL environment variable is set or a file named OA.DATAMODEL is found in the data directory of your OpenAccess installation hierarchy. See the discussion Specifying a DataModelRev for a Flow for more information.
-DMAttributes attributeList Specifies a list of DM attributes that are used for the new library that the translator creates. Attributes are specified as name=value pairs. There must be no space on either side of the equal sign, two or more pairs must be space-separated, and the list must be enclosed in quotes. For example: 
    verilog2oa ... -DMAttributes "name1=value name2=value name3=value"
-DMSystem sysName Name of the DM library system for the translated library, such as
oaDMFileSys for compatibility with previous releases or oaDMTurbo for the Turbo DM system
-help | -h Prints help
-libDefFile filename Specifies the lib.defs file to load. The path to the lib.defs file can be a full or relative path. If this option is omitted, the default search mechanism of the DM system is used to find and load the lib.defs file.
-libPath path Path to which the destination library is written
-logFile file Name of the log file. The default is oa20to22.log in the current working directory.
-noInfo msgIds Suppresses the specified INFO messages. msgIds is a quoted, space separated list of numbers. Each number in the list represents the numerical portion of the ID for the message you want to suppress. None of the numbers in the list may be zero. Suppressed messages do not appear on the terminal or in the log file.
-noWarning msgIds Suppresses the specified WARNING messages. msgIds is a quoted, space separated list of numbers. Each number in the list represents the numerical portion of the ID for the message you want to suppress. None of the numbers in the list may be zero. Suppressed messages do not appear on the terminal or in the log file and are not included in the total of WARNING messages displayed in the summary.
-overwrite Replaces an existing design with the translated data. If this option is omitted, the default behavior is to update the existing library.
-shared Allows other applications to access the output library while this application is running.
-templateFile file

Specifies a file containing arguments to oa20to22. You can specify a template file instead of entering a string of arguments on the command line.

If you specify a template file, arguments on the command line have precedence over arguments specified in the file. So, if the same argument exists in the template file and in the command line, the translator uses the argument on the command line.

Specify arguments in a template file as follows:

  • Enter arguments in the template file without a dash (-) before the argument.
  • Enter each argument and value pair on a single line.
  • Separate the argument from its value using a space or a tab.
  • Designate comment lines with a # sign as the first character in the line.

Sample Template File

# oa20to22 command line arguments

logFile        myoa20to22.log
libPath        /usr/tbd/mylib
-version | -v Prints version information

Command line arguments specifying library, cell, and view names are interpreted according to the Native name space. See Native Name Space in the Name Mapping article of the Programmers Guide for details about the native name space.

Note: If you want to update client source code to reflect changes to the object names in OpenAccess 2.2, refer to Migrating Your 2.0 Applications to OpenAccess 2.2.

Updates Performed by oa20to22

The following modifications occur during the translation.

Technology

Replace RouteSpecs with ConstraintGroups

  • oaRouteSpecs in your design are replaced by oaConstraintGroups of the same name. The following tables outlines the changes.

    2.0 Value 2.2 Simple Constraint Type
    viaDefArray oacValidRoutingVias
    layerSpecArray oacValidRoutingLayer

  • The width and spacing values (both regular and diagonal) for each layer in the layerSpecArray become corresponding constraints on the layer. The isHard attributes for layer constraints are set according to the isWidthFixed and isSpacingFixed values of the layer spec.

    2.0 Layer Value in the layerSpecArray 2.2 Layer Constraint Type
    layer width in layerSpecArray oacMinWidth
    layer spacing in layerSpecArray oacMinSpacing
    diagWidth in layerSpecArray oacMinDiagonalWidth
    diagSpacing in layerSpecArray oacMinDiagonalSpacing

  • Single layer rules are translated into oaLayerConstraints, as follows.

    2.0 Rule Type 2.2 Constraint Type
    oacMinAreaRule oacMinArea
    oacMinEnclosedAreaRule oacMinEnclosedArea
    oacMinDensityRule oacMinDensity
    oacMaxDensityRule oacMaxDensity
    oacMinEdgeLengthRule oacMinEdgeLength`
    oacMinSpaceRule oacMinSpacing
    oacMinDiagonalSpaceRule oacMinDiagonalSpacing
    oacMinDiffPontentialSpaceRule oacMinDiffPotentialSpacing
    oacMinEnclosedSpaceRule oacMinEnclosedSpacing
    oacMinFillPatternSpaceRule oacMinFillPatternSpacing
    oacMinProximitySpaceRule oacMinProximitySpacing
    oacMinProtudedProximitySpaceRule oacMinProtrudedProximitySpacing
    oacMinSameNetSpaceRule oacMinSameNetSpacing
    oacMinAdjacentViaSpaceRule oacMinAdjacentViaSpacing
    oacMergeSpaceAllowedRule oacMergeSpaceAllowed
    oacMinWidthRule oacMinWidth
    oacMinDiagonalWidthRule oacMinDiagonalWidth
    oacMinProtrusionWidtRule oacMinProtrusionWidth
    oacMaxWidthRule oacMaxWidth
    oacMinNumCutRule oacMinNumCut
    oacMinProtrusionNumCutRule oacMinProtrusionNumCut
    oacDefaultAntennaRule Default model in oacAntenna
    oacSecondAntennaRule Second model in oacAntenna
    oacPeakACCurrentDensityRule Converted into layer attribute
    oacAvgACCurrentDensityRule Converted into layer attribute
    oacRmsACCurrentDensityRule Converted into layer attribute
    oacAvgDCCurrentDensityRule Converted into layer attribute
    oacMinButtedAreaRule Obsolete

  • Double layer rules are translated into oaLayerPairConstraints. The rule type and constraint type mapping is as follows.

    2.0 Rule Type 2.2 Constraint Type
    oacMinClearanceRule oacMinClearance
    oacMinSameNetClearanceRule oacMinSameNetClearance
    oacMinExtRule oacMinExtension
    oacCoincidentExtRule oacCoincidentExtensionAllowed
    oacMinOverlapRule oacMinOverlap
    oacViaStackingAllowedRule oacViaStackingAllowed
    oacMinOppositeExtRule oacMinDualExtension
    oacMinButtedClearanceRule Obsolete
  • The 2.0 RouteSpec associations for oaNets and oaRoutes are replaced by constraint groups with the same names that are members of the constraint group for the oaNets or oaRoutes.
  • The via stack limits on a technology database are translated into oaSimpleConstraints of constraint type oacViaStackLimit. The constraint value is the maximum number of vias allowed in the stack, and the parameters are the top and bottom layer of the stack.
  • All constraints created in the technology database belong to the foundry rules constraint group.
  • The isError attribute in oaDerivedLayer is converted to a layer constraint of type oacErrorLayer on the derived layer.
  • The routing grid pitch and offset on physical layers are translated into oaLayerConstraints according to their preferred routing direction.

    Preferred Routing 2.2 Direction Constraint Type
    oacRouteDirHorz      

    oacVerticalRouteGridPitch
    oacVerticalRouteGridOffset
    oacRouteDirVert      

    oacHorizontalRouteGridPitch    
    oacHorizontalRouteGridOffset
    oacRouteDirLeftDiag   

    oac45RouteGridPitch                      
    oac45RouteGridOffset

    oacRouteDirRightDiag

    oac135RouteGridPitch                      
    oac135RouteGridOffset

  • When one of the side gate or diode area ratios is non-zero in a 2.0 oaAntennaRule, both the side ratios values are used in the new 2.2 oaAntennaRatioValue, and the isSide attribute is set to true. Otherwise, the regular gate and diode area ratios are used and the isSide attribute is set to false.
  • The cumulative antenna area ratios for via and metal on a technology database are translated into oaSimpleConstraints of constraint type oacCumViaAntenna and oacCumMetalAntenna. The default and second antenna models become the corresponding values in the oaAntennaRatioArray for these constraints. oa20to22 only creates these constraints when one of the area ratios is non-zero. The isSide attribute is set on the 2.2 antenna ratio value when either of the ratio’s isSide attributes is set on the 2.0 cumulative ratio values.

ViaSpec Changes

  • viaSpec now uses oaViaDef2DTblValue to contain the width and viaDefArray information. Only one viaSpec is allowed for a layer pair combination. All 2.0 viaSpecs with the same layer pair combination are consolidated into a single 2D lookup table for the oaViaDef2DTblValue. The minWidth and maxWidth for layer 1 are in the rowHeader, and the minWidth and maxWidth for layer 2 are in the colHeader of the 2D lookup table. The viaDefArray is appended to each entry with the corresponding row and column header for the layer widths. When there is no range specified for a 2.0 viaSpec, it is appended to the default viaDefArray of the corresponding 2.2 viaSpec.

Rule to Constraint Conversion

  • Each 2.2 oaConstraint type has a set of allowed value types. When the value of a 2.0 rule is not one of the allowed value types in its corresponding constraint type, an error is issued and the rule is not translated.
  • The application attribute extensions on a 2.0 oaRule are translated into extensions on the corresponding 2.2 oaConstraint.

Route Changes

  • The shielding net information has been moved from oaRoute to oaShape and oaVia in 2.2. The shielding net information is copied from the 2.0 route to each oaPathSeg and oaVia in the new 2.2 route.
  • An oaRouteSeg is translated to an oaPathSeg on a layer purpose pair determined by the layer number of the oaRouteSeg and oaRoutePurpose. The route purpose's oaRoutePurposeDrawing is mapped into oavPurposeNumberDrawing, and oaRoutePurposeFill is mapped into oavPurposeNumberFill.

Blockage and Boundary Changes

  • A placement blockage is translated into an oaAreaBlockage, and other types of blockages are translated into oaLayerBlockages.
  • oaCellViewBoundary objects are renamed to oaAreaBoundary objects

Guide Changes

In 2.2, an oaGuide is created with start and end points instead of start and end conn objects. The translator uses the 2.0 start conn object to derive a start point, and the 2.0 end conn object to derive the end point. Only shapes, routes, instances, terminals, instance terminals, pins, and instance pins are handled in this manner. For other objects, the location of the guide defaults to (0, 0).

Object 2.2 Guide Placement
Shapes The guide is placed on the center point of the bounding box.
Routes The guide is placed on the end point of the route.
Terminal The guide is placed on the center of the bounding box of the first pin's figure if it contains pins, otherwise (0, 0) is used.
Instance terminal

The guide is placed on the center point of the first pin on the instance terminal if the instance is bound and the terminal has a pin, otherwise the guide is placed on the instance origin.

Pin The guide is placed on the center of the bounding box of the pin's figure.
Instance Pin The guide is placed on the center of the instance pin if the instance is bound, otherwise the guide is placed on the instance origin.
All other objects The guide defaults to (0,0)

Steiner Changes

In 2.2, an oaSteiner contains a bounding box and can be associated with a layer. When translating 2.0 versions of oaSteiner and oaSteinerPin, no layer is used during creation. The bounding box is used if it is a steiner pin, otherwise, (0, 0) is used.

Connectivity

  • Pins that do not belong to a pin group are interpreted as mustJoins in 2.2. For a pin that has no pin group in a terminal that contains multiple pins or pin groups, a new net and terminal are created, and the new terminal is set as a mustJoin terminal to the original terminal.
  • Instance pins are not translated.
  • When a pin's figure is a route in 2.0, the route is not converted, but its elements are converted and attached to the 2.2 pin. This implies that all elements in the route become strongly connected.
  • PinGroups are eliminated. For oaStrongPin group, a single pin is created and all the pin figures are added to this pin. For oaWeakPin group, the relationship is implied by pins on the same terminal.

Properties and Extensions

  • An oaDDObjectDB attached to a library is translated into an oaDMData object by copying its properties and application extension objects. When an oaDDObjectDB is attached to a cell or a view in a cell, its corresponding oaDMData is created in the oaCell and oaCellView objects, respectively. The groups on oaDDObjectDBs are copied.
  • Properties on oaCellviews are transferred to oaDesigns.

Pcells

After translation, a Pcell instance is created with a standard design as the master. The expansion of the Pcell is triggered when the design is read. If the design is subsequently saved, the expanded state is also saved and expansion will not be triggered again.

For more information about Pcell expansion, refer to Conversion Between Pcells and Standard oaDesigns in the Support for Pcells section of the Programmers Guide.

Miscellaneous

  • Values that an application sets on a 2.0 oaCMap are not translated to the 2.2 oaCMap.
  • Pseudo hierarchy names are not allowed in 2.2. When they are used as instance or net names, they are converted by replacing ‘/' with “#2f”.
  • oaMarker severity values are mapped to oaMarkerSeverity values as follows.

    oaMarker Severity Value 2.2 oaMarkerSeverity Value
    0 oacAnnotationMarkerSeverity
    1 oacInfoMarkerSeverity
    2 oacAcknowledgeWarningMarkerSeverity
    3 oacWarningMarkerSeverity
    4 oacWarningMarkerSeverity
    5 oacErrorMarkerSeverity
    6 oacSignedOffCriticalErrorMarkerSeverity
    7 oacSignedOffCriticalErrorMarkerSeverity
    8 oacFatalErrorMarkerSeverity
    Greater than 8 oacInfoMarkerSeverity
